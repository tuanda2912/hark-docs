## [2026-06-28] traceability | applied the feature→file framework → [[feature-map]]
Added a **feature → service → subsystem** traceability layer ([[feature-map]]) — the "if I change feature X, which files move?" map. Hark is the **microservices test instance** of a feature↔file mapping framework being prototyped (a monolith instance was built first against an unrelated repo): it's a real **polyglot multi-process** app, so it exercises the cross-service case a single-language monolith can't. The page records: (1) the **3-service partition** (`harkd` Swift / `main` Node / `renderer` Angular) derived from the graph's 7 layers, with each subsystem assigned to its process; (2) the **cross-process contracts** — the WS [[subsystems/wire-protocol]] (`harkd`↔`renderer`, snake_case JSON hand-mirrored in `engine.service.ts`) and the [[subsystems/preload-security]] IPC (`renderer`↔`main`) — flagged because **no shared compiler crosses the Swift↔TS seam**, so a wire-frame change won't fail the renderer build (only `RagWireTests.swift` pins the engine side); (3) a **~10-row feature register** (live transcription · diarization · vault RAG/Ask · summary+Q&A · translation · vault/git · privacy/egress · onboarding · tray · packaging) → services + subsystems + status, grounded in [[STATUS|STATUS.md]] (Phase 6 shipped; Phase 7 packaging ~60%). Key insight recorded: Hark's code-shaped wiki already supplied `subsystem → files` (the `## Code map` blocks), so **subsystem = capability tag** and the framework only had to add the feature register + process partition + contracts. The one open gap surfaced: **no renderer-side wire contract test** (STATUS open thread 7) — the hand-mirror is unchecked across the seam. Graph fresh @ `69d53bc` (4 files behind HEAD `8e7d009`); no re-run needed. Catalogued under *Overview* in `index.md`.
